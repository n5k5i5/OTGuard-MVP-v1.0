# Шлюз согласования организации (OrgGate)

Механизм требует ввода данных компании/организации и явного подтверждения уполномоченным лицом перед использованием системы. Цель — обеспечить законное, этичное и авторизованное применение.

## Как работает

- Инициализация через `framework org init`:
  - Обязательные поля: `--name`, `--domain`, `--email`
  - Токен подтверждения генерируется, его SHA-256 хэш сохраняется в `.runs/org.json`.
  - Если настроен SMTP, отправляется email; иначе токен выводится в CLI.
  - Опциональный вебхук: если задан `ORG_WEBHOOK_URL`, отправляется JSON на указанный адрес.

- Подтверждение:
  - `framework org verify --token <TOKEN>`
  - Кликабельная ссылка (опционально):
    - Запустите локальный HTTP-сервер:
      ```bash
      framework org serve --host 0.0.0.0 --port 8080
      ```
    - Установите `ORG_VERIFY_BASE_URL=http://<host>:8080/verify`, чтобы вставлять ссылку в письма.

- Статус/Сброс:
  - `framework org status` → текущая запись (хэш токена маскируется).
  - `framework org reset --confirm` → очищает запись.

## Принудительное применение

- После принятия EULA основные команды (modules, run, resource, report, sessions и т. п.) блокируются до подтверждения OrgGate.
- Разрешены до подтверждения: `help`, `init`, `org *`, `about`.

## SMTP / Вебхук

- Переменные окружения SMTP:
  - `SMTP_HOST`, `SMTP_PORT` (по умолчанию 587), `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` (по умолчанию `SMTP_USER`), `SMTP_TLS` (по умолчанию `true`)
  - `ORG_VERIFY_BASE_URL` (опционально; кликабельная ссылка в письмах)
- Вебхук:
  - `ORG_WEBHOOK_URL` (опционально; JSON POST при init)

## Хранение данных

- `.runs/org.json`: запись об организации и хэш токена.
- `token_hash` маскируется в выводе статуса.

## Заметки по безопасности

- Храните токен в секрете, чтобы избежать несанкционированного подтверждения.
- Данные SMTP/вебхука храните безопасно (env/secret manager).
- Дизайн fail-closed: без подтверждения критические команды заблокированы.
